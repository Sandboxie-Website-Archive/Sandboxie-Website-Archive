Hello Axie,
(I am not native in English idiom)

I had exactly the same problem as described by you.
And the problem occurs when Web Anti Virus feature is enabled. You can check it by deactivating that in Kaspersky and try run Chrome inside Sandboxie.

But how to fix that? You should be asking yourself.

I solved this by putting a clause inside Web Antivirus feature, telling it not to check the address http://www.google.com.br (in my case is br, since I am from Brazil). To do that follow these steps:

1. Open Kaspersky Panel
2. Click over Settings (bottom left of the main window)
3. Click over Protection Tab
4. Find and click over Web Anti-Virus
5. A new Windows will open. Make sure to enable this feature sliding the green key at the right top of the windows
6. Click on Advanced Settings
7. Now you have to put the google address inside two links: The link Configure Exclusions and the link Configure Trusted Web Pages. But pay attention: here is a trick: you must remove http:// ou https:// from the address in order it can work. I suffered to find out that. Kaspersky seems to ignore the adress when you put the protocols before the address


I hope it can solve your issue the same way it solved to me.



Elias

Ive had same exact problem on a handful of machines using IE11. I have more than 2 dozen running sandboxie and KIS2015 and first encountered the problem months back upgrading machines from 2014 to 2015 KIS. Long story short the fix for me is the version of KIS installed and you need newest sandboxie (4.16 coming soon). I found using version kis15.0.0.463en_6024 so far has solved the problem. I have more machines to "fix" tomorrow and ill post back success rates.

Uninstall KIS using the uninstaller or the removal tool in safe mode if needed. Reboot.

Install the build kis15.0.0.463en_6024. Start the program and manually update it, will be a large update. This upgrades you to 463(b). Reboot again.

Now see if you still have the problem.


kis15.0.0.463en_6024 SHA-1:586C2F92E12048A5826266A0AC01F7D86C04BA61
MD5:9054FAA83B918EDB1387FECE2D2E781A

Small Update:
The correct Extension to Core communication can be enabled by this rules:
OpenPipePath=*chrome.nativeMessaging.*
OpenIpcPath=\RPC Control\epmapper

Can be easily checked with DbgView ,-) Extension working fine now.

But this will not fix the SSL warning issue which seems to be a different problem. This does not seem a problem related to Pipes or IPC/RPC as even * exclusion do not help. Will try to enable Kaspersky trace-logging for some more info....
(See post aboves for workarounds for the SSL-bug. My workaround is Kaspersky->Setting->Advanced->Trusted Applications->Add "Chrome.exe" and set "do not scan encrypted traffic".)
